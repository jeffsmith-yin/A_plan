# 融智桥 · 技能包与智能解决方案商业化方案（购物车 + 分成）

> **文档性质**：产品 / 商业化 **Draft（草案）**，供评审与迭代。
> **版本**：v0.2（2026-07-11）　**关联**：已交付 4（ChainAdapter 多签法币）/ 已交付 5（法律模板）/ 已交付 7（团队知识库 §3.2 Skills 封装、§3.4 贡献激励）/ 已交付 8（AI 智能体专家 MVP 原型，本方案已落地为代码）。
> **核心命题**：把专家/智能体沉淀的方法论固化为**可复用技能包**，把正式交付客户的**智能解决方案**也做成可售卖单元，二者均可**加入购物车**，并**按创建人（自然人或智能体）拟定分成**，沿用「透明智能分账协议」。
> **v0.2 修订（用户澄清）**：技能包**实际可用 = 购买 + 平台协助注入真实数据**；缺真实数据则全为演示数据，即使持有技能包文件/截屏也不可用。新增 §七 真实数据注入与激活机制（含防泄露 DRM-by-design、PIPL/数据处理合规、分账里程碑拆分）。

---

## 一、目标
1. 专家/智能体的方法论 → **可复用、版本化、可售卖**的技能包资产；
2. 面向客户的正式交付 → **智能解决方案**（技能包组合 + 定制），同样可加入购物车；
3. 分账**按创建人**计（自然人专家 或 AI 智能体），与已交付的数字解决方案分成方案一致。

## 二、技能包（Skills Pack）
- **定义**：可复用、版本化的专家方法论资产 = 规则模板 + 方法步骤 + 预警指标。
- **关键属性**：`id` / `name` / `domain` / `version` / `creator{type:'person'|'agent', name}` / `listPrice` / `share`。
- **创建人**：
  - **自然人专家**（35+，假名）：其授权方法论封装；
  - **AI 智能体**：由智能体专家沉淀、经审核的方法论自动封装（见已交付 7 §3.6、已交付 8）。
- **复用性**：同一技能包可售给多个客户——一次创作、多次交付，显著降低边际成本（呼应「AI 粘性 + 规模交付」）。

## 三、智能解决方案（正式交付客户）
- **组成**：若干技能包组合 + 客户定制规则（来自 DEMO 闯关中外确定的约束）。
- **交付单元**：里程碑 + DEMO 游戏化验收（见 PRD §4）→ 触发分账。
- 方案本身也是购物车中的可售卖单元（可含定制溢价）。

## 四、购物车
- 客户将**技能包 / 智能解决方案**加入购物车，统一结算；
- 原型已落地（`agent-expert/`）：CLI 自动加购 + Web `/cart/add`、`/cart/checkout`；
- 生产对接：购物车 → 订单 → 持牌托管清分（见已交付 5）。

## 五、分成方案（按创建人）
- **原则**：技能包 / 解决方案按**创建人（自然人或智能体）**拟定分成，与「数字解决方案」分成方案**同一套**——透明智能分账协议。
- **标准拆分**（落在已交付 5 两段式费率区间内）：

| 接收方 | 比例 | 说明 |
|--------|------|------|
| 创建人（自然人/智能体） | ~80% | 100% − 平台 − 链上；自然人→该专家，智能体→其运营/AI 人才方 |
| 平台 take rate | 18% | 区间 10–20% |
| 链上清算费 | 2% | 区间 1–3% |

- **结算机制**：2-of-3 多签（专家/企业/平台仲裁）+ **人民币法币**（联盟链测试网模拟，预留真实 TBaaS/蚂蚁链，见已交付 4 `ChainAdapter`）。
- 与 PRD §4.5、法律模板（已交付 5）完全对齐；禁止公链代币/稳定币（币链分离）。

## 七、技能包真实数据注入与激活机制（用户澄清，2026-07-11）

### 7.1 核心规则
> **技能包默认都是演示数据；实际可用必须满足两步：① 使用者「购买」（下单结算）；② 「厂家在平台的协助下注入真实数据」后，技能包才「激活可用」。**
> 否则即便拿到技能包文件或截屏，**没有真实数据，技能包都不能用**。

- 演示态（DEMO、未注入真实数据）的技能包：全部为演示数据，仅用于走查/验收，**不产生真实交付效力**，不可用于实际生产决策。
- 激活态：购买完成 + 平台协助注入客户真实数据 → 技能包绑定该客户真实数据 → 真正可用。

### 7.2 生命周期（新增「激活」阶段）
```
创建(专家/智能体方法论) → 版本化/上架 → 加购 → 购买结算(触发2-of-3多签法币分账)
        → 【平台协助注入真实数据】 → 激活可用 → 持续运维 / 版本升级
```
- 购买结算与"真实数据注入"是**两个可拆分里程碑**（见 7.5 分账衔接）。

### 7.3 防泄露 / DRM-by-design（重要推论）
- 技能包**静态资产（文件/截屏）本身不可用**——价值绑定在「平台协助注入的真实数据 + 平台运行环境」，而非资产副本。
- 因此外泄技能包文件/截屏**风险极低**：拿不到真实数据 + 无平台注入通道 = 无法运行。这天然弱化了"卖课/立门派"式泄密与侵权动机，与运营方案（已交付 6）数字人+假名、法律模板（已交付 5）内容授权/NDA **同源互补**。
- 平台仍须守住三道闸：
  1. **NDA + 内容授权**（已交付 5）约束真实数据的使用范围与再分发；
  2. **真实数据注入通道权限管控**——仅平台/授权运维可操作，禁开放给普通成员；
  3. **链上存证审计链**——记录「谁 / 何时 / 为哪家客户 / 注入了哪份真实数据」，可追溯、可争议举证。

### 7.4 PIPL / 数据安全（新增合规项）
- 真实企业数据属**客户资产**；平台"协助注入"构成**受托数据处理**——须签**数据处理协议（DPA）**、最小必要、目的限定、留痕可审计；
- 真实数据**不出企业域、不进公开链**（币链分离；链上仅存证不存资金）；脱敏/加密存储；
- 与已交付 5（PIPL/PIA）、已交付 7（知识权属、NDA/内容授权）联动；跨境数据另行评估。

### 7.5 与分账里程碑衔接
- **购买结算**：触发「创建人~80% / 平台 18% / 链上 2%」的透明智能分账（§五、已交付 5）；
- **真实数据注入 = 交付关键里程碑**，建议设为「验收/尾款分账」触发点（与 PRD §4 里程碑 + DEMO 验收触发分账一致）：
  - 首付（购买时）：结算部分款项；
  - 验收（注入激活后）：结算尾款，平台完成交付确认。
- 两段式分账既可保障创建人收益，也给企业客户"注入激活后才付全款"的验收安全感。

### 7.6 对 MVP（已交付 8）的影响
- 当前 `agent-expert` 仅演示数据；生产须增加**「真实数据注入适配层」**：客户上传/系统对接 → 平台权限校验 → 绑定技能包 → 激活；并配套权限与审计（见 §九 待细化）。

## 八、合规
- 全部演示数据；链上仅存证不存资金；
- AI 智能体专家/技能包创建人身份清晰标注，不冒充真实自然人（与数字人+假名同源）；
- 智能体创建人的收益归属须在协议中明确（运营/AI 人才方），避免权属纠纷；
- **真实数据受托处理须签 DPA、最小必要、留痕可审计**（见 §7.4）；真实数据不出企业域、不进公开链。
