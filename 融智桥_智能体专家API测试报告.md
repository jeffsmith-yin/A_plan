## 三、安全评估

### 3.1 已通过（不崩溃 / 正确拦截）
- **SQL 注入**（`'; DROP TABLE skills; --`）：返回 200，词法检索不触达任何数据库，无注入面；不崩溃。✅
- **XSS**（`<script>alert(1)</script>`）：载荷作为纯文本保留在 JSON 字段中，未被执行；前端页面用 `textContent` 渲染，无 DOM XSS。✅
- **畸形 JSON**：映射为 **400**（`readJson` 统一解析）。✅（修复前会返回 500 甚至挂起）
- **超大负载**（> 1 MB）：触发 **413** 体限，防 DoS。✅（修复前无上限）

### 3.2 标记缺口（MVP 阶段可接受，生产必须补齐）
| 项 | 现状 | 风险 | 生产建议 |
|----|------|------|----------|
| 鉴权 | 所有端点无 Bearer Token | 任意调用者可触发分析与结算 | 网关/中间件加 JWT，结算类端点强制鉴权 |
| 限流 | 无（120 并发请求全部放行，无 429） | 易被滥用/刷量 | 网关层按 IP/Token 限流（如 60 req/min） |
| 传输 | 仅 HTTP（演示） | 中间人篡改 | 正式环境强制 HTTPS/TLS |
| 购物车隔离 | 单进程内存、全局共享 | 多用户串号 | 按 session/user 隔离 + 持久化（接已交付 4 的 `uni.*` 存储或后端） |
| 审计日志 | 无请求/结算留痕 | 分账争议难追溯 | 结算走链上存证 + 服务端审计日志（接已交付 5 透明智能分账协议） |
| 输入 Schema | 仅校验 ID 存在性 | 字段类型/范围未约束 | 加 JSON Schema 校验 |

## 四、测试中发现并修复的缺陷（已交付 8/9 → 10 回馈）

| # | 严重度 | 缺陷 | 根因 | 修复 |
|---|--------|------|------|------|
| 1 | 🔴 高 | `POST /cart/add` 对**不存在的技能包 ID 返回 `ok:true`**，幽灵条目进入购物车，结算时被静默丢弃 | `addToCart` 未校验技能是否存在 | `cart.js` 改为先查知识库 `kb.skills`，不存在即 `return false` |
| 2 | 🔴 高 | **畸形 JSON 导致服务挂起**（客户端 8s 超时） | `readJson` 的 `responded` 标志在解析前被置真，`fail(400)` 提前 return 不发响应 | 仅在解析**成功**后置 `responded=true`；解析失败走 `fail(400)` 正常回 400 |
| 3 | 🟡 中 | 请求体**无大小上限**（5 MB 负载被全盘读入，DoS 风险） | 原始实现无体积约束 | `readJson` 加 1 MB 体限，超限回 **413** 并 `req.destroy()` |

> 上述修复已随本测试一并落地于 `src/cart.js`、`src/server.js`，并通过回归（20/20）。

## 五、问题、风险与优化建议

### 5.1 上线前必须（P0）
1. **鉴权 + 限流**：结算与加购端点强制 JWT；网关层按 Token/IP 限流，缺失即拦。
2. **购物车隔离与持久化**：从全局内存改为按用户 session 隔离，并持久化（多端一致，接 Uni-app 存储或后端）。
3. **结算审计**：每次 `checkout` 落服务端审计日志 + 联盟链存证，支撑分账争议追溯（已交付 5）。

### 5.2 建议优化（P1–P2）
- **检索升级**：`rag.js` 词法匹配 → embedding 向量检索（语义召回更准，已交付 7 §3.3）。
- **生成升级**：规则拼接 → LLM + 行业 Skills 提示词，输出更自然可控。
- **多域扩展**：当前仅制造业，扩至零售/餐饮/外贸等域，每域一个智能体专家。
- **输入校验**：引入 JSON Schema 对 `text`/`id` 做类型与范围约束。
- **可观测性**：加请求日志、错误率、P95 延迟埋点，接已交付 4 数据看板。

### 5.3 已固化资产
- `test/api-test.js`：零依赖、可重复、可作为 CI 验收门禁（`npm test`，退出码 0=通过 / 1=失败 / 3=超时兜底）。
- 单次运行 < 1 秒、无外部依赖，易于接入流水线。
