# 融智桥 · 团队组建与知识库规划

> **文档性质**：项目协调 / 技术负责人 **Draft（草案）**，供评审与迭代。
> **版本**：v0.1　**适用阶段**：DEMO 收尾 → MVP 启动（衔接已交付 1–6）
> **定位**：正式产品路径第二阶段——在「市场调研 → PRD → DEMO/跨端/法律/运营」之后，进入**团队/知识库 → 交付 → 运维/验收**。
> **核心命题**：用**知识库**驱动「不同阶段的 AI 专家自动介入」，把 35+ 专家方法论、AI 人才工程力、企业需求、平台撮合粘合成可规模化交付的系统。

---

## 一、目标与原则

| 维度 | 目标 |
|------|------|
| 团队 | MVP 期可运行的最小可行团队（~12–16 人核心 + 外部专家网络） |
| 知识库 | 沉淀行业方法论 / Skills / 模板 / 案例，成为 AI 自动介入的"大脑" |
| 协同 | 四方（专家 / AI 人才 / 企业 / 平台）在线协作 + 阶段性 AI 自动介入 |
| 原则 | 阶段化介入、知识驱动 AI、最小可行、合规优先（PIPL / 币链分离 / 演示数据） |

## 二、团队组建

### 2.1 角色全景（四类参与方 + 平台职能 + 自动介入 AI）
| 角色 | 职责 | 介入阶段 | 来源 |
|------|------|----------|------|
| 行业专家（35+） | 供给方法论、定义规则/约束、验收把关 | DEMO 起持续 | 种子库（目标 30） |
| AI 技术人才 | Agent/数据/工程化、把规则变成可运行方案 | DEMO 起持续 | 种子库（目标 20） |
| 企业代表 | 提需求、参与 DEMO、游戏化验收 | DEMO 起持续 | 运营获客 |
| 平台方 | 撮合、规则、仲裁、托管对接、运营 | 全周期 | 自建 |
| **自动介入 AI 专家** | 按阶段注入的 AI 助手（需求/开发/测试/合规/验收） | 各阶段 | 知识库 + Agent 编排 |

### 2.2 平台核心职能团队（MVP 编制建议）
| 职能 | 编制 | 优先级 | 说明 |
|------|------|--------|------|
| 产品 / PM | 1–2 | P0 | 需求拆解、里程碑、四方协调 |
| UI/UX | 1 | P0 | Uni-app 跨端体验（见已交付 4） |
| 前端（Uni-app） | 1–2 | P0 | 微信小程序 / H5 / App |
| 后端 / 平台 | 2 | P0 | 入驻、IM、看板、积分、推荐 |
| 联盟链 / 清算工程 | 1–2 | P1 | ChainAdapter 真实化（TBaaS/蚂蚁链，见已交付 4/5） |
| 算法 / AI 工程 | 1–2 | P1 | 知识库 RAG、Agent 编排、数字人管线 |
| QA / 验收 | 1 | P1 | BUG 跟踪、验收用例（见 §四） |
| 运营 / 增长 | 1 | P0 | 种子招募 + 短视频矩阵（见已交付 6） |
| 内容 / 数字人 | 1 | P1 | 数字人脚本与产出（已定数字人+假名出镜） |
| 法务 / 合规 | 外包+兼职 | P0 | 协议终审、PIPL（见已交付 5） |

> 总编制 **~12–16 人** 即可启动 MVP；专家/AI 人才走外部网络（种子库 + 顾问制），不计入全职。

### 2.3 分阶段介入时间表（衔接 PRD §8）
| 阶段 | 介入角色 | 关键动作 |
|------|----------|----------|
| DEMO 期 | 产品+设计+前端+后端+法务(模板)+运营(种子) | 闯关原型、NDA、种子招募 |
| MVP 期 | + 联盟链工程 + QA + 算法 | 真实分账对接、知识库 RAG、自动介入 Agent |
| 全功能 | + 安全 + 数据 + 海外本地化 | 多端发布、跨境合规、环比看板满 1 年 |

### 2.4 四方协同机制
- **专家（35+）**：以「授权方法论 + 数字人/假名呈现」贡献（见已交付 6），不直接写代码；
- **AI 人才**：承接「专家定规则 → AI 配 Agent → 企业验收」工程化；
- **企业**：提痛点、走 DEMO、游戏化验收、触发分账；
- **平台**：撮合 + 2-of-3 仲裁 + 托管对接；
- **自动介入 AI 专家**：在对应阶段把知识库内容注入工作流（见 §3.5）。

### 2.5 组织与协作
- 远程优先，使用融智桥自身 IM（三方实时交流）作为协作场；
- 双周迭代（Sprint），里程碑制（对齐 PRD §7 阶段划分）；
- 轻量 RACI：产品定 What、技术定 How、专家定 Domain、企业定 Accept、平台定 Rule；
- 周对齐会：专家×AI 人才，持续明确各角色介入时间表（PRD §8）。

### 2.6 招募与激励
- 核心团队：自建 + 关键岗位外包（联盟链/合规）；
- 外部专家网络：接运营种子库（30 专家 + 20 AI），顾问制 + 分账激励；
- 贡献激励：方法论/代码/案例贡献计入平台积分（新成员 +10 / 推荐 +2 去重），高价值 Skills 可享调用分成（待定）。

## 三、知识库规划

### 3.1 定位：让「AI 专家自动介入」成真的"大脑"
知识库不只是文档归档，而是 **RAG 语料 + 技能包（Skills）+ 模板** 的复合体。不同阶段/场景的 AI 助手从知识库检索对应内容，自动介入工作流——这正是项目规范要求的「不同平台、不同阶段由不同 AI 专家自动介入」。

### 3.2 知识分类体系
| 知识域 | 内容 | 主要贡献方 | 权限 |
|--------|------|------------|------|
| 行业方法论 / Skills | 专家可交付方法论、规则模板、行业 Know-how | 专家（35+） | 内部（脱敏） |
| DEMO 案例库 | 制造业排产等闯关案例、验收记录 | 企业+AI | 内部 |
| 分账规则模板 | 2-of-3 多签、两段式费率规则 | 平台+法务 | 机密 |
| 合规与合同 | 联盟链+法币协议、NDA、PIPL | 法务 | 机密 |
| 产品文档 | PRD、竞品分析、路线图 | 产品 | 内部 |
| 运营素材 / 数字人脚本 | 短视频脚本、矩阵打法 | 运营+内容 | 内部 |
| 工程规范 | Uni-app、ChainAdapter、API 规范 | 技术 | 内部 |
| 验收 / BUG 用例 | 验收标准、缺陷库 | QA | 内部 |

### 3.3 技术架构（草案）
- **文档层**：Markdown / Wiki（版本化、可评审），已交付 1–6 文档作为**种子语料**直接入库；
- **向量层**：行业方法论 / Skills / 合规要点做 embedding，支撑语义检索；
- **编排层**：按阶段路由到对应 Agent（需求助手 / 开发助手 / 测试助手 / 合规助手 / 验收助手）；
- **隔离**：DEMO 与正式数据严格分离，知识库内容须**脱敏 / 演示化**，不混入真实个人数据（PIPL）。

### 3.4 权限与治理
- **分级权限**：公开 / 内部 / 机密（分账与合规为机密）；
- **版本与审核**：专家署名 + 法务/技术复核后方可入向量库；
- **贡献激励**：积分 + 高价值 Skills 调用分成（与 §2.6 一致）；
- **质量**：定期评审、过时下架、冲突合并。

### 3.5 与「AI 自动介入」的映射
| 阶段 / 场景 | 自动介入 AI 专家 | 调用知识 |
|-------------|------------------|----------|
| 需求分析 | 竞品/PRD 助手 | 竞品分析、PRD、行业痛点 |
| 方案设计 | 专家方法论推荐 | 行业 Skills、规则模板 |
| 开发 | 代码/工程助手 | 工程规范、ChainAdapter 接口 |
| 企业使用 DEMO | 行业数字人 / 技能包推荐 | 专家方法论、DEMO 案例 |
| 交付 | 验收助手 | 验收用例、BUG 库 |
| 运维 | BUG 跟踪助手 | 缺陷库、历史修复 |
| 合规 | 合规助手 | 联盟链+法币协议、NDA、PIPL |

### 3.6 落地里程碑
| 周次 | 动作 | 产出 |
|------|------|------|
| W1–2 | 搭知识库骨架（Wiki + 向量库），导入已交付 1–6 为种子语料 | 知识库 v0.1 |
| W3–4 | 行业 Skills 首批准入（接种子专家 30） | 制造业等首批 Skills |
| W5–8 | 接入自动介入 Agent（需求/开发/验收） | 阶段化 AI 介入 MVP |

## 四、交付 / 运维 / 验收衔接（下一子阶段）
- **交付**：以里程碑 + DEMO 闯关为交付单元，2-of-3 多签触发分账（见已交付 5）；
- **运维 / BUG 跟踪**：缺陷库入知识库，BUG 跟踪助手自动归类与指派；
- **验收测试**：游戏化验收（企业代表）+ QA 用例库（见 §3.2/§3.5）；
- 全程演示数据隔离，真实上线前完成 PIA（个人信息影响评估）。

---

## 五、风险与合规
- **PIPL**：知识库个人数据最小化，专家名片/方法论脱敏，演示数据不入库真实信息；
- **知识权属**：专家方法论授权范围须与 NDA / 内容授权（数字人+假名）一致，明确可调用与分成边界；
- **合规红线**：链上仅存证不存资金、禁公链代币（见已交付 5）；
- **依赖风险**：自动介入 AI 不得越权代签合同 / 代触发真实分账，关键动作留人工确认。
